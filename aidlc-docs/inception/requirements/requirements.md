# 要件定義書: マークダウンエディター

## 意図分析サマリー

| 項目 | 内容 |
|---|---|
| **ユーザーリクエスト** | ブログ記事執筆に最適な、最高のマークダウンエディターを自作したい。CMS/note/zennへの直接アップロード機能付き。軽量に実現。 |
| **リクエストタイプ** | 新規プロジェクト（Greenfield） |
| **スコープ** | システム全体（フルアプリケーション） |
| **複雑度** | 高（WYSIWYG編集、プラグインシステム、マルチプラットフォーム連携、ハイブリッドアプリ） |
| **要件深度** | 包括的（Comprehensive） |

---

## 機能要件

### FR-01: アプリケーション形態
- **ハイブリッド構成**: Tauri + Web技術
- Webフロントエンドを共有し、デスクトップ版（Tauri）とWeb版（ブラウザ）の両方を提供
- デスクトップ版: ローカルファイルへの直接アクセス、ネイティブな操作感
- Web版: インストール不要、ブラウザからアクセス可能

### FR-02: WYSIWYG編集体験
- Notion/Typoraライクなリッチテキスト編集
- Markdownソースを意識させないシームレスな編集体験
- インライン書式（太字、斜体、コード、リンク等）のリアルタイムレンダリング
- ブロック要素（見出し、リスト、引用、コードブロック、テーブル等）の視覚的編集

### FR-03: 拡張可能なMarkdownサポート
- **ベース**: GitHub Flavored Markdown（GFM）
- **プラグインシステム**: カスタム構文の追加が可能
  - 数式（KaTeX/MathJax）
  - ダイアグラム（Mermaid等）
  - 脚注
  - 各プラットフォーム独自の構文（Zennのメッセージ記法等）
- プラグインは遅延読み込み（起動速度への影響を最小化）

### FR-04: プラットフォーム連携
- **対象プラットフォーム（初期）**:
  - note
  - Zenn
  - microCMS
  - Contentful
- **連携方式**:
  - API連携: エディターから直接投稿（公開/下書き保存）が可能なプラットフォーム
  - エクスポート: APIが無い/制限のあるプラットフォーム向けにフォーマット変換してコピー/ダウンロード
- **プラットフォームアダプター**: 各プラットフォームの連携をプラグインとして実装し、将来的な拡張を容易にする

### FR-05: 画像管理
- プラットフォーム連携時に、対象プラットフォームの画像ストレージに自動アップロード
- エディター内でのドラッグ&ドロップ画像挿入
- 画像のプレビュー表示（WYSIWYG内）

### FR-06: ファイル管理
- **プロジェクト/ワークスペースベース**
  - フォルダツリー表示
  - 複数ファイルの管理
  - ブログ記事の一覧管理
- ファイルの新規作成、開く、保存、名前変更、削除
- 最近開いたファイルの履歴

### FR-07: エクスポート機能
- Markdown（.md）
- HTML
- 各プラットフォーム固有のフォーマット（プラットフォームアダプターによる変換）

## 非機能要件

### NFR-01: パフォーマンス（起動速度重視）
- **起動速度**: アプリケーションが瞬時に立ち上がること（目標: 1秒以内）
- プラグインの遅延読み込みにより、コア機能の起動を高速化
- エディターのレンダリングはスムーズかつ遅延なし
- 大きなMarkdownファイル（10,000行以上）でも快適に編集可能

### NFR-02: 技術スタック（推奨）
- **フロントエンド**: 軽量さを最優先で選定（推奨: Svelte/SvelteKit — バンドルサイズが小さく、ランタイムオーバーヘッドが最小）
- **デスクトップ**: Tauri（Rust製、Electronよりも大幅に軽量）
- **エディターエンジン**: ProseMirror or Tiptap（WYSIWYGに最適、拡張性が高い）
- **言語**: TypeScript

### NFR-03: アクセシビリティ
- 基本的なアクセシビリティ対応
  - キーボードナビゲーション（すべての主要機能にキーボードでアクセス可能）
  - スクリーンリーダー対応（適切なARIAラベル）
  - フォーカス管理

### NFR-04: 拡張性
- プラグインアーキテクチャにより、以下を後から追加可能:
  - 新しいMarkdown構文
  - 新しいプラットフォーム連携
  - 新しいエクスポート形式
  - UIテーマ

## 設計上の考慮事項

### 起動速度 vs プラグインシステムのバランス
- プラグインの遅延読み込み（lazy loading）を必須とする
- コア機能（エディター、ファイル管理）は即座に利用可能にし、プラグイン（プラットフォーム連携、拡張構文）はバックグラウンドまたはオンデマンドで読み込む
- プラグインマニフェストは軽量なJSONで管理し、実際のコードは使用時にのみロード

### ハイブリッドアプリの構成
- Web版とデスクトップ版で同一のフロントエンドコードベースを共有
- プラットフォーム固有の機能（ファイルシステムアクセス等）はアダプターパターンで抽象化
  - Web版: File System Access API / ブラウザのファイル選択
  - デスクトップ版: TauriのファイルシステムAPI

### プラットフォーム連携のAPI制約
- note: 公式APIの制約を確認の上、可能な範囲で対応（APIが制限的な場合はエクスポート方式）
- Zenn: GitHub連携ベースのワークフロー対応を検討
- microCMS / Contentful: REST APIによる直接連携

## セキュリティ
- セキュリティ拡張ルール: **スキップ**（プロトタイプ/実験的プロジェクトとして開発）
- 基本的なセキュリティ対策（APIキーの安全な保存、XSS防止）は常識的な範囲で実施
